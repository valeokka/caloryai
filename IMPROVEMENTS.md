# Улучшения бота - Новые возможности

## Реализованные улучшения

### 1. ✅ Кэширование результатов по file_id

**Что это дает:**
- Экономия на запросах к OpenAI API (не платим за повторный анализ одного фото)
- Мгновенный ответ при повторной отправке того же фото
- Снижение нагрузки на API

**Как работает:**
- При получении фото бот проверяет, анализировалось ли оно ранее (по file_id)
- Если да и кэш не устарел (по умолчанию 24 часа) - возвращает сохраненный результат
- Пользователь видит пометку "♻️ Это фото уже анализировалось ранее"
- Можно отключить через `CACHE_ENABLED=false` в .env

**Настройки:**
```env
CACHE_ENABLED=true          # включить/выключить кэш
CACHE_TTL_HOURS=24         # время жизни кэша в часах
```

**Файлы:**
- `src/database/queries/requests.js` - функция `getCachedResult()`
- `src/bot/handlers/photo.js` - проверка кэша перед анализом
- `src/config/constants.js` - настройки CACHE

### 2. ✅ Логирование стоимости запросов к OpenAI

**Что это дает:**
- Мониторинг расходов на OpenAI API в реальном времени
- Понимание, сколько стоит каждый запрос
- Возможность оптимизации (например, уменьшить max_tokens)

**Что логируется:**
- Количество использованных токенов (prompt + completion)
- Стоимость запроса в долларах
- Время выполнения запроса
- Модель, которая использовалась

**Пример лога:**
```
INFO: OpenAI response received {
  usage: {
    promptTokens: 1234,
    completionTokens: 89,
    totalTokens: 1323
  },
  cost: "$0.0198",
  duration: "2341ms"
}
```

**Обновление цен:**
Цены OpenAI указаны в `src/services/openai.js` в константе `PRICING`. Обновляй их при изменении тарифов:

```javascript
const PRICING = {
  'gpt-4o': {
    input: 0.005,   // $ за 1K токенов
    output: 0.015   // $ за 1K токенов
  }
};
```

**Файлы:**
- `src/services/openai.js` - расчет стоимости в `_calculateCost()`

### 3. ✅ Ограничение размера фото (max 5MB)

**Что это дает:**
- Защита от слишком больших файлов
- Экономия на передаче данных
- Более быстрая обработка

**Как работает:**
- Проверка размера фото перед обработкой
- Если фото больше лимита - пользователь получает понятное сообщение
- Показывается фактический размер фото и максимально допустимый

**Сообщение пользователю:**
```
⚠️ Размер фото слишком большой. Максимальный размер: 5МБ.
Попробуй сжать фото или отправить другое.

Размер твоего фото: 8.3МБ
```

**Настройка:**
```env
MAX_PHOTO_SIZE_MB=5    # максимальный размер в МБ
```

**Файлы:**
- `src/bot/handlers/photo.js` - проверка размера
- `src/config/constants.js` - настройки VALIDATION.PHOTO

### 4. ✅ Улучшенная обработка ошибок OpenAI

**Что это дает:**
- Понятные сообщения для пользователей в зависимости от типа ошибки
- Правильная обработка разных сценариев (rate limit, quota, auth и т.д.)
- Логирование типа ошибки для мониторинга

**Типы ошибок и сообщения:**

| Ошибка | Сообщение пользователю | Когда возникает |
|--------|------------------------|-----------------|
| Timeout | ⚠️ Сервис анализа временно недоступен | Превышено время ожидания |
| Rate Limit | ⚠️ Слишком много запросов к AI. Попробуй через минуту | Превышен лимит запросов в минуту |
| Auth (401) | ⚠️ Сервис анализа временно недоступен | Неверный API ключ |
| Quota (402) | ⚠️ Сервис временно недоступен | Закончились деньги на счету OpenAI |
| Content (400) | ⚠️ Не удалось обработать это изображение | Фото нарушает политику контента |
| Unknown | ⚠️ Произошла ошибка при обработке запроса | Другие ошибки |

**Умный retry:**
- Не ретраим ошибки аутентификации (401) и контента (400)
- Ретраим только временные ошибки (timeout, rate limit)
- Экспоненциальная задержка между попытками

**Файлы:**
- `src/services/openai.js` - функция `_handleOpenAIError()`

## Статистика и мониторинг

### Просмотр логов стоимости

```bash
# Все запросы с стоимостью
pm2 logs calorie-bot | grep "cost"

# Только успешные запросы
pm2 logs calorie-bot | grep "OpenAI response received"

# Ошибки OpenAI
pm2 logs calorie-bot | grep "Error in analyzeFood"
```

### Анализ кэша

```bash
# Попадания в кэш
pm2 logs calorie-bot | grep "Cache hit"

# Все запросы (с кэшем и без)
pm2 logs calorie-bot | grep "Successfully processed photo"
```

### SQL запросы для аналитики

```sql
-- Статистика по кэшу (сколько раз использовалось одно фото)
SELECT photo_file_id, COUNT(*) as usage_count
FROM requests_history
GROUP BY photo_file_id
HAVING COUNT(*) > 1
ORDER BY usage_count DESC;

-- Средняя стоимость запросов (если добавить поле cost в БД)
-- Можно расширить таблицу для хранения стоимости

-- Самые популярные блюда
SELECT dish_name, COUNT(*) as count
FROM requests_history
GROUP BY dish_name
ORDER BY count DESC
LIMIT 10;
```

## Рекомендации по оптимизации

### Снижение расходов на OpenAI

1. **Уменьшить max_tokens** (если ответы всегда короткие):
   ```env
   OPENAI_MAX_TOKENS=300
   ```

2. **Увеличить TTL кэша** (если пользователи часто отправляют одни фото):
   ```env
   CACHE_TTL_HOURS=72
   ```

3. **Использовать более дешевую модель** (если точность не критична):
   ```env
   OPENAI_MODEL=gpt-4o-mini
   ```

### Мониторинг расходов

Создай скрипт для подсчета дневных расходов из логов:

```bash
# Примерный скрипт
pm2 logs calorie-bot --lines 10000 | \
  grep "cost" | \
  grep "$(date +%Y-%m-%d)" | \
  awk '{print $NF}' | \
  sed 's/[^0-9.]//g' | \
  awk '{sum+=$1} END {print "Total: $" sum}'
```

## Тестирование

### Проверка кэша

1. Отправь фото в бота
2. Отправь то же фото еще раз
3. Второй раз должен быть мгновенный ответ с пометкой "♻️"

### Проверка размера фото

1. Отправь фото больше 5МБ
2. Должно прийти сообщение об ошибке с указанием размера

### Проверка обработки ошибок

Для тестирования можно временно:
- Указать неверный API ключ (проверка auth error)
- Установить очень маленький timeout (проверка timeout error)

## Миграция

Изменения обратно совместимы. Для применения:

```bash
pm2 restart calorie-bot
```

Если хочешь отключить кэш:
```bash
echo "CACHE_ENABLED=false" >> .env
pm2 restart calorie-bot
```

## Дополнительные улучшения (опционально)

### Сохранение стоимости в БД

Можно добавить поле `api_cost` в таблицу `requests_history`:

```sql
ALTER TABLE requests_history 
ADD COLUMN api_cost DECIMAL(10, 6) DEFAULT NULL,
ADD COLUMN api_tokens INTEGER DEFAULT NULL;
```

Затем обновить `createRequest()` для сохранения этих данных.

### Dashboard для мониторинга

Можно создать простой endpoint для просмотра статистики:
- Общая стоимость за день/неделю/месяц
- Количество кэш-попаданий
- Самые дорогие запросы
- Средняя стоимость запроса

### Алерты

Настроить уведомления при:
- Превышении дневного бюджета на OpenAI
- Частых ошибках API
- Низком проценте кэш-попаданий
